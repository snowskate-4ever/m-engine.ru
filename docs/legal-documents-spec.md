# ТЗ: раздел «Юридические документы»

Обновлено: 2026-04-17

## Цель

Добавить в музыкальную экосистему единый раздел юридических документов для публичных сущностей, чтобы:

- владелец профиля мог управлять реквизитами и юридическими файлами в кабинете;
- модерация могла проверять публикацию до выхода в публичную зону;
- на публичной странице отображались только одобренные и явно разрешенные к публикации документы.

## Контекст и зависимости

- Базовая модель профилей, публичных страниц и решений по данным: [music-ecosystem-plan.md](./music-ecosystem-plan.md).
- Текущий контур модерации и админ-флоу: [music-ecosystem-roadmap.md](./music-ecosystem-roadmap.md).
- Механика отображения в публичном профиле должна использовать существующие `layout_draft` и `layout_published`.
- Политики доступа и роли должны расширять существующие owner/admin/moderation сценарии, без дублирования архитектуры.

## Термины

- **Юридический документ**: сущность с типом, статусом модерации, видимостью и полезной нагрузкой (структурированные поля и/или файл).
- **Версия документа**: снимок состояния документа при публикации/изменении с датами действия.
- **Публичный документ**: версия документа со статусом `approved` и `visibility = public`.

## Scope MVP (этап 1)

### 1) Объекты, для которых доступен раздел

Раздел доступен в кабинете владельца для:

- `Shop`
- `School`
- `Studio`
- `Rehersal`
- `RecordLabel`
- `ProducerCenter`
- `Teacher` (опционально; включается флагом конфигурации)

### 2) Типы документов

MVP поддерживает:

- `entity_details` - структурированные реквизиты субъекта;
- `public_offer` - оферта;
- `privacy_policy` - политика конфиденциальности;
- `return_policy` - условия возврата;
- `delivery_policy` - условия доставки;
- `other` - иной юридический документ.

### 3) Статусы и видимость

- Статусы жизненного цикла: `draft`, `pending_review`, `approved`, `rejected`, `archived`.
- Видимость: `owner_only`, `public`.
- В публичную выдачу попадают только версии, удовлетворяющие обоим условиям:
  - `status = approved`;
  - `visibility = public`.

### 4) Версионирование

- Любое изменение публикационной версии формирует новую запись версии.
- У версии фиксируются `version`, `effective_from`, `effective_to`.
- В документе хранится ссылка на текущую актуальную версию.

### 5) Модерация

- Отправка на модерацию переводит документ в `pending_review`.
- Модератор может:
  - одобрить (`approved`);
  - отклонить (`rejected`) с обязательной причиной.
- Лог модерации сохраняется в аудит-сущности.

## Scope этапа 2 (после MVP)

- Шаблоны документов по типу сущности и юридическому статусу.
- Правила обязательности документов для отдельных типов профилей (например, магазин без оферты не может публиковать витрину).
- Фиксация юридических акцептов пользователями (кто/когда/какой версией согласился).
- Отдельный операторский UI очереди модерации с SLA и эскалациями.

## Модель данных

## Основные таблицы

### `legal_documents`

Назначение: карточка документа, текущий статус и связь с владельцем.

Минимальные поля:

- `id`
- `owner_type`, `owner_id` (polymorphic)
- `document_type` (enum)
- `title`
- `status` (enum)
- `visibility` (enum)
- `current_version_id` (nullable FK на версии)
- `reviewed_by` (nullable FK users)
- `reviewed_at` (nullable datetime)
- `rejection_reason` (nullable text)
- `created_by` (FK users)
- `updated_by` (nullable FK users)
- `created_at`, `updated_at`, `deleted_at` (soft delete по решению команды)

Индексы:

- `idx_legal_documents_owner` (`owner_type`, `owner_id`)
- `idx_legal_documents_public_filter` (`status`, `visibility`, `document_type`)

### `legal_document_versions`

Назначение: хранение версий документа с фактическим содержимым.

Минимальные поля:

- `id`
- `legal_document_id` (FK)
- `version` (integer, > 0)
- `payload_json` (json; структурированные поля и метаданные)
- `file_path` (nullable string; путь в storage)
- `external_url` (nullable string; если используется ссылка вместо файла)
- `checksum` (nullable string)
- `effective_from` (datetime)
- `effective_to` (nullable datetime)
- `published_by` (nullable FK users)
- `published_at` (nullable datetime)
- `created_at`, `updated_at`

Индексы:

- unique (`legal_document_id`, `version`)
- `idx_legal_document_versions_effective` (`effective_from`, `effective_to`)

### `legal_document_audits` (или расширение существующей audit-модели)

Назначение: трассировка изменений и модерационных действий.

Поля:

- `id`
- `legal_document_id`
- `action` (`created`, `updated`, `submitted`, `approved`, `rejected`, `archived`, `published`)
- `old_values` (json)
- `new_values` (json)
- `performed_by` (FK users)
- `performed_at` (datetime)

## Правила валидации

- Для `entity_details` требуется `payload_json` с реквизитами по `legal_entity_type`.
- Для файловых типов требуется хотя бы одно: `file_path` или `external_url`.
- Для `external_url` разрешены только `https`.
- При переходе в `approved` у версии обязателен `effective_from`.
- `rejection_reason` обязателен при `rejected`.

## API и маршруты

Ниже предлагается контракт уровня HTTP. Точные namespace и middleware подтверждаются при реализации.

### Кабинет (owner/admin)

- `GET /api/music/legal-documents/{ownerType}/{ownerId}`
  - список документов владельца;
  - фильтры: `status`, `document_type`, `visibility`.
- `POST /api/music/legal-documents/{ownerType}/{ownerId}`
  - создание документа.
- `GET /api/music/legal-documents/{id}`
  - карточка документа + список версий.
- `PATCH /api/music/legal-documents/{id}`
  - обновление черновика.
- `POST /api/music/legal-documents/{id}/versions`
  - создание новой версии (draft).
- `POST /api/music/legal-documents/{id}/submit`
  - отправка на модерацию.
- `POST /api/music/legal-documents/{id}/archive`
  - архивирование документа.

### Модерация (MoonShine/moderator)

- `POST /api/music/legal-documents/{id}/approve`
  - одобрение документа/версии.
- `POST /api/music/legal-documents/{id}/reject`
  - отклонение с причиной.

### Публичная зона

- `GET /api/public/{entityType}/{slug}/legal-documents`
  - read-only список одобренных публичных документов.
- Для SSR/Blade публичных страниц применяется тот же фильтр (`approved + public`) в серверном сервисе.

## Политики доступа

- **Owner**: CRUD своих документов, submit на модерацию, чтение всех версий.
- **Moderator/Admin**: просмотр всех документов, approve/reject/archive.
- **Guest/User**: только read-only публичной выдачи одобренных документов.
- Запрещено отдавать `rejection_reason`, внутренние аудит-данные и черновые версии в публичном API.

## UI/UX требования

## Кабинет владельца

Экран «Юридические документы» в карточке сущности:

- список документов с бейджами статуса и видимости;
- быстрые действия: «Редактировать», «Отправить на модерацию», «Архивировать»;
- блок версий: номер, дата начала действия, автор, текущая/архивная.

Форма документа:

- тип документа, заголовок, видимость;
- структурированные поля реквизитов (для `entity_details`);
- загрузка файла или ввод ссылки;
- предпросмотр публикационной версии;
- отображение причины отклонения и подсказки, что исправить.

## MoonShine (модератор)

- ресурс `LegalDocumentResource` с фильтрами по:
  - `status`;
  - `document_type`;
  - типу владельца (`owner_type`);
  - дате отправки/проверки.
- карточка модерации с действиями `approve/reject`.
- обязательное поле причины отклонения при reject.

## Публичная страница

- блок `legal_documents` в настройках layout-блоков (вкл/выкл, порядок).
- отображается компактный список:
  - наименование документа;
  - дата вступления в силу;
  - кнопка просмотра/скачивания.
- если нет документов для показа, блок скрывается без ошибки.

## Нефункциональные требования

- Аудит действий обязателен для всех переходов статусов.
- Файлы должны храниться в изолированном диске/префиксе, ссылки - с контролем доступа.
- Публичная выдача не должна раскрывать внутренние поля модерации.
- Нужны тесты на позитивные и негативные сценарии прав доступа.

## Набор тестов (минимум)

### Feature

- создание документа владельцем;
- отправка на модерацию;
- одобрение и отклонение модератором;
- публичная выдача только `approved + public`;
- создание новой версии при изменении.

### Policy

- запрет доступа к чужим документам для owner;
- допуск модератора к approve/reject;
- запрет публичного чтения черновиков/отклоненных.

### Integration/UI

- отображение статусов и причин отклонения в кабинете;
- корректная работа блока в публичной странице через layout.

## Критерии приемки MVP

- Владелец сущности создает документ и отправляет его на модерацию.
- Модератор одобряет или отклоняет документ с причиной.
- На публичной странице отображаются только одобренные публичные документы.
- Изменение опубликованного документа создает новую версию.
- Политики доступа и ключевые сценарии покрыты тестами.

## Out of scope для MVP

- Юридически значимая подпись (КЭП/ЭЦП).
- Автоматическая генерация текста документов на основе шаблонного конструктора.
- Интеграции со сторонними провайдерами комплаенса.
