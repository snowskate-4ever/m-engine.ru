# План ручного тестирования интерфейсов и страниц

Обновлено: 2026-04-17

## Цель

Проверить, что ключевые пользовательские интерфейсы работают стабильно после изменений: страницы открываются, действия выполняются, ограничения доступа соблюдаются, ошибки отображаются корректно.

## Формат выполнения

- Каждый кейс содержит: **ID**, **что проверяем**, **шаги**, **ожидаемый результат**.
- Выполняем блоками, чтобы можно было тестировать параллельно разными людьми.
- Результат кейса фиксировать в формате: `PASS` / `FAIL` / `BLOCKED` + короткий комментарий.

## Тестовые роли (минимум)

- `Гость` (неавторизованный пользователь).
- `Пользователь` (обыный авторизованный).
- `Владелец музыкального профиля` (может редактировать свой профиль/карточку).
- `Владелец магазина` (имеет доступ к инвентарю и заказам своего магазина).
- `Админ/оператор` (доступ к административным страницам).

## Параллельные блоки

### Поток A (публичная зона, без логина)

Независим от остальных потоков.

- Блок A1: Главная, discover, публичные профили.
- Блок A2: SEO/технические страницы (`robots`, `sitemap`).

### Поток B (личный кабинет и коммуникации)

Требует авторизации, можно идти параллельно потоку C и D.

- Блок B1: Дашборд, уведомления, календарь.
- Блок B2: Мессенджер и настройки уведомлений.
- Блок B3: Настройки профиля/безопасности.

### Поток C (музыкальные сущности и каталог)

Требует пользователя с правами на редактирование сущностей.

- Блок C1: Профили и карточки (создание/редактирование).
- Блок C2: Поиск заявок и discover в кабинете.
- Блок C3: Жалобы на публичные профили.

### Поток D (магазин)

Лучше разделить на две роли: покупатель и владелец магазина.

- Блок D1: Витрина/корзина/оформление заказа.
- Блок D2: Кабинет магазина (инвентарь, заказы, шаблон импорта).

### Поток E (служебные разделы для администраторов)

Требует админ-роль.

- Блок E1: Ручной запуск подбора исполнителей.
- Блок E2: Интеграция с VK (токены, группы, ленты, сбор постов).
- Блок E3: Доски задач и история изменений.

## Как это должно работать (кратко для входа в задачу)

Это раздел для быстрого понимания системы. Перед выполнением тестов новый участник читает его 5-7 минут, чтобы понимать не только "куда нажать", но и "зачем это нужно".

### Публичная часть (Поток A)

- Пользователь без логина должен видеть витрину сервиса: главную страницу, каталог (`discover`) и публичные профили по `slug`.
- Переходы между публичными страницами должны быть прямыми и предсказуемыми: без авторизации, без неожиданных редиректов на личный кабинет.
- Технические страницы (`robots`, `sitemap`) нужны для поисковиков и внешней индексации, поэтому всегда должны открываться и содержать валидные данные.

### Личный кабинет и коммуникации (Поток B)

- После входа пользователь попадает в рабочее пространство: дашборд, уведомления, календарь, сообщения.
- Уведомления и мессенджер должны сохранять состояние: если сообщение отправлено или настройка изменена, после обновления страницы данные не теряются.
- Настройки профиля и безопасности должны быть безопасными по умолчанию: ошибки валидации понятны, критичные действия (пароль, 2FA) подтверждаются.

### Музыкальные карточки и каталог (Поток C)

- Сервис хранит разные типы сущностей (исполнители, студии, школы, магазины и т.д.), но пользовательский сценарий должен быть единым: создать, отредактировать, опубликовать.
- Пользователь может редактировать только свои сущности; попытка зайти в чужую карточку должна блокироваться.
- Публичный контент модерируется через жалобы: форма жалобы должна отправляться, а защита от спама (ограничение частоты) - срабатывать.

### Магазин (Поток D)

- Для покупателя сценарий простой: найти товар -> добавить в корзину -> оформить заказ -> увидеть заказ в истории.
- Для продавца отдельный сценарий: управлять остатками и заказами своего магазина.
- Данные должны быть согласованы между ролями: то, что оформил покупатель, должно появиться у владельца магазина.

### Админские и служебные разделы (Поток E)

- Это внутренние инструменты операций: ручной запуск подбора, страницы VK-интеграции, доски задач.
- Их задача - поддержка процессов, поэтому ключевая проверка: доступность, понятные ошибки, корректное сохранение действий оператора.
- В этих разделах важнее стабильность и диагностируемость, чем "красота" интерфейса.

## Быстрый сценарий онбординга тестировщика (15-20 минут)

1. Прочитать раздел "Как это должно работать".
2. Получить тестовые доступы по ролям (гость, пользователь, владелец магазина, админ).
3. Пройти по одному кейсу из каждого потока A-E (smoke), чтобы увидеть общую картину.
4. После этого брать свой поток полностью и фиксировать результаты в формате `PASS/FAIL/BLOCKED`.

## Блок A1. Публичные страницы каталога

### A1-01 Главная страница

- Что проверяем: открытие landing и базовые блоки.
- Шаги:
  1. Открыть `/`.
  2. Проверить наличие блоков каталога/категорий.
  3. Перейти по 1-2 ссылкам из блоков.
- Ожидаемый результат: главная загружается без ошибок, ссылки ведут на корректные страницы.

### A1-02 Общий discover

- Что проверяем: доступность публичного каталога.
- Шаги:
  1. Открыть `/discover`.
  2. Применить фильтр/поиск (если доступно в UI).
  3. Открыть карточку из результатов.
- Ожидаемый результат: список обновляется, карточка открывается, 500/404 не появляется.

### A1-03 Discover по категориям

- Что проверяем: scoped discover.
- Шаги:
  1. Открыть валидную категорию `/discover/musicians` (или другую существующую).
  2. Открыть заведомо невалидную категорию `/discover/invalid-category`.
- Ожидаемый результат: валидная страница открывается; для невалидной показывается 404.

### A1-04 Публичный профиль по slug

- Что проверяем: публичный профиль открывается по `slug`.
- Шаги:
  1. Открыть профиль по маршруту вида `/musicians/{slug}` или `/public/{slug}`.
  2. Проверить отображение основных данных и контактов.
- Ожидаемый результат: профиль корректно отображается, структура страницы не ломается.

## Блок A2. SEO и технические страницы

### A2-01 robots.txt

- Что проверяем: корректная выдача robots.
- Шаги:
  1. Открыть `/robots.txt`.
  2. Проверить наличие ссылки на sitemap.
- Ожидаемый результат: plain text, присутствует строка `Sitemap: .../sitemap-music.xml`.

### A2-02 sitemap

- Что проверяем: генерация sitemap.
- Шаги:
  1. Открыть `/sitemap-music.xml`.
  2. Проверить, что XML валиден и содержит URL.
- Ожидаемый результат: отдается XML без ошибок, в списке есть discover/публичные страницы.

## Блок B2. Мессенджер

- Что проверяем: загрузка workspace.

### B2-02 Отправка сообщения

- Что проверяем: отправка и отображение нового сообщения.
- Шаги:
  1. В открытом диалоге отправить сообщение.
  2. Обновить страницу.
- Ожидаемый результат: сообщение сохраняется и остается в истории после перезагрузки.

### B2-03 Настройки уведомлений мессенджера

- Что проверяем: сохранение пользовательских настроек.
- Шаги:
  1. Открыть `/messenger/settings/notifications`.
  2. Изменить 1-2 параметра.
  3. Сохранить и обновить страницу.
- Ожидаемый результат: новые значения сохраняются и повторно отображаются.

## Блок B3. Настройки аккаунта и безопасность

### B3-01 Профиль

- Что проверяем: редактирование профиля пользователя.
- Шаги:
  1. Открыть `/settings/profile`.
  2. Изменить поля профиля.
  3. Сохранить.
- Ожидаемый результат: сохранение успешно, изменения видны после обновления.

### B3-02 Пароль

- Что проверяем: смена пароля с валидациями.
- Шаги:
  1. Открыть `/settings/password`.
  2. Проверить негативный сценарий (неверный текущий пароль).
  3. Проверить позитивный сценарий.
- Ожидаемый результат: при ошибке показывается валидация; при успехе пароль обновляется.

### B3-03 2FA и внешний вид

- Что проверяем: доступность страниц настроек.
- Шаги:
  1. Открыть `/settings/two-factor`.
  2. Открыть `/settings/appearance`.
- Ожидаемый результат: страницы доступны, переключатели/элементы интерфейса работают без JS-ошибок.

## Блок C1. Музыкальные профили и карточки

### C1-01 Раздел профилей

- Что проверяем: общая точка входа.
- Шаги:
  1. Открыть `/profiles`.
  2. Переключить вкладки типов профилей.
- Ожидаемый результат: вкладки переключаются, контент обновляется корректно.

### C1-02 Создание карточки исполнителя

- Что проверяем: сценарий create.
- Шаги:
  1. Открыть `/music/performers/create`.
  2. Заполнить обязательные поля.
  3. Сохранить карточку.
- Ожидаемый результат: карточка создается, после сохранения доступна для редактирования/просмотра.

### C1-03 Редактирование своей карточки

- Что проверяем: update + права.
- Шаги:
  1. Открыть `/music/performers/{id}/edit` для своей записи.
  2. Изменить данные и сохранить.
  3. Попробовать открыть чужую запись без прав.
- Ожидаемый результат: свои изменения сохраняются; для чужой записи — запрет доступа.

### C1-04 Карточки площадок и организаций

- Шаги:
  1. Для 1-2 типов пройти `index -> create -> edit`.
  2. Проверить одинаковое поведение базовых полей (адрес, описание, публикация).
- Ожидаемый результат: структура сценария единообразна, сохранение работает, ошибки валидации читаемы.

## Критерии готовности релиза по ручному тесту

- Нет `FAIL` в блоках A1, B1, C1, D1 (критичный пользовательский путь).
- Критичные операции (логин, редактирование профиля, отправка сообщений, оформление заказа) проходят минимум в одном полном цикле.
- Ошибки авторизации и валидации отображаются пользователю понятным текстом, без необработанных исключений.
- Для каждого `FAIL` заведена задача с шагами воспроизведения и скриншотом.

## Что фиксировать при дефекте

- Роль пользователя и URL страницы.
- Шаги воспроизведения (коротко, по пунктам).
- Факт vs ожидание.
- Скриншот/видео и время проверки.
- Частота: стабильно / плавающе.
