# Дальнейшее развитие музыкального сервиса (дорожная карта)

Обновлено: 2026-04-11.  
Документ дополняет закрытый чеклист в `music-ecosystem-plan.md` (п.1–11) и фиксирует **следующие эпики** без обязательства к срокам.

---

## Контекст

Базовый контур экосистемы реализован: публичные профили, кабинет, состав исполнителей с приглашениями, каталог для авторизованных, адреса, дублирование приглашений на e-mail. Ниже — направления, которые усиливают продукт для пользователей и операций.

### Что логично делать следующим (Phase 4+ vs MoonShine)

- **Магазины (фазы 4–7)** — основной продуктовый горизонт, если приоритет — монетизация и оффер для продавцов. Объём большой: сущности, остатки, корзина, заказы, оплата.
- **MoonShine / админка (раздел 3 roadmap, фаза 9)** — имеет смысл **раньше или параллельно узкой полосой**, если уже есть инциденты с данными, ручная модерация или нужно сопровождать импорты и «битые» slug без SQL. Полноценная админка под заказы имеет смысл **после** появления сущностей магазина (как в фазе 9: «по мере готовности»).
- **Качество и наблюдаемость (раздел 6)** — не конкурирует с двумя ветками: логи почты/пригласительных сценариев и метрики можно наращивать **параллельно** любыми фазами.

Итого: при чётком продуктовом «да» на коммерцию — **ведущая нить фазы 4+**; **минимальный MoonShine** (просмотр музыкальных сущностей, выкл. публикации, slug) стоит закладывать, как только операционка начинает болеть. Полный MoonShine под каталог заказов — ближе к фазам 5–6.

## 1. Центр in-app уведомлений

**Статус (2026-04-09): реализовано** — см. детализацию в разделе «Фаза 1» ниже.

**Зачем:** записи в `notifications` уже создаются (например, приглашение в состав), но без единой точки входа пользователь может их не замечать.

**Состав работ (изначально):**

- UI в шапке / боковой зоне: счётчик непрочитанных, выпадающий список последних событий.
- Страница «Все уведомления» с пагинацией, фильтром по типу (по `type` / данным payload).
- Пометка прочитано / «прочитать всё»; при ответе на приглашение — сброс в `PerformerMembershipService` + broadcast синхронизации для открытых вкладок.
- Локализация заголовков и текста по типам уведомлений (в т.ч. `PerformerLineupInvitationNotification`).

**Зависимости:** нет жёстких; можно делать параллельно с другими пунктами.

## 2. Публичный каталог (без обязательной авторизации)

**Статус (2026-04-09):** реализован маршрут `GET /discover` (`route('discover')`) с `MusicDirectoryPage` и `MusicPublicSearchService`; в кабинете по-прежнему `music/discover` (`music.discover`) для авторизованных.

**Дальнейшая доработка (по желанию):**

- Расширенный SEO (отдельные лендинги под типы, sitemap).
- Кэширование или индексация для больших каталогов.

**Статус (2026-04-11):** маршруты `GET /discover/{category}` (`discover.category`) с отдельными title/description/canonical; `GET /sitemap-music.xml` (`MusicSitemapController`) — discover, витрины по типам и публичные slug-профили; динамический `GET /robots.txt` со ссылкой на sitemap; кэш 120 с для `MusicPublicSearchService::publicCatalogCounts()`; **сброс кэша** при создании/изменении `public_page_enabled`, `slug`, `deleted_at` и при удалении/восстановлении (`PublicMusicCatalogCacheObserver` на публичных музыких сущностях). В тестах по-прежнему можно вызывать `forgetPublicCatalogCountsCache()` явно.

## 3. Администрирование в MoonShine (или отдельный admin)

**Зачем:** операционная поддержка, модерация «сухим» способом, обход проблемных записей без SQL.

**Статус (2026-04-10):** в MoonShine добавлены ресурсы магазинов (`ShopResource`, `ShopItemResource`, `ShopOrderResource`, `ShopOrderItemResource`), публичных музыких сущностей (`app/MoonShine/Resources/MusicEcosystem/*`), меню разбито на группы «Музыка · справочники / магазины / публичные профили». У `AddressResource` расширен `MorphTo` владельца (музыкант, учитель, исполнитель, студия, репточка, школа, магазин, лейбл, продюсерский центр). Строки заказа (`ShopOrderItemResource`) в меню не выводятся — только просмотр из заказа; создание заказов из админки отключено; в форме заказа — статус исполнения, **оплата (суммы, комиссия, статус/способ без PSP)**; позиции заказа не редактируются из админки (снимок при оформлении). В формах магазина и музыких профилей — **модерация** и у магазина **ставка комиссии**; на индексе — дата скрытия модерацией. Подсказки к полям JSON вёрстки — `App\MoonShine\Support\MusicAdminHints`.

**Роли:** доступ к разделам MoonShine по-прежнему задаётся учётками и ролями MoonShine (см. ресурс ролей в админке). Отдельная роль «поддержка» оформляется выдачей прав на нужные ресурсы в этом интерфейсе, без связки с `Gate` приложения.

**Состав работ (оставшееся / улучшения):**

- Ресурсы или расширение существующих: `Musician`, `Teacher`, `Peformer`, `Studio`, `Rehersal`, `School`, `Shop`, `RecordLabel`, `ProducerCenter` — просмотр, мягкое отключение публикации, правка slug при конфликтах.
- Связанные `Address` для музыких сущностей (полиморфный владелец уже есть в модели `Address`; в MoonShine MorphTo может потребовать расширения типов).
- Роли: доступ только админам MoonShine, без смешения с пользовательскими политиками `Gate`.

**Статус (2026-04-11):** ресурс `PublicProfileReportResource` (жалобы пользователей); в заказах магазина в форме MoonShine — режим доставки и адрес.

**Зависимости:** решение, что правится только из админки, а не дублирует весь кабинет.

## 4. Интеграции и дополнительные каналы уведомлений

**Зачем:** в спецификации п.8 заложены «чат, уведомления» и расширение на контакт; сейчас есть сайт + БД + почта.

**Возможные направления:**

- **Мессенджер:** при приглашении создавать системное сообщение / карточку в существующем чате (если есть общий канал с пользователем) — продуктово сложнее, нужны правила (только при существующем диалоге или всегда «бот»).
- **Telegram / VK:** опциональная привязка аккаунта и отправка короткого уведомления (в проекте уже есть зачатки VK у пользователя — оценить переиспользование).
- Единый **шлюз уведомлений** (какие каналы включены в `notification_preferences` наряду с `music_lineup_email`).

**Зависимости:** политика приватности, лимиты внешних API, очереди.

**Зависимости:** юридические формулировки, политика хранения причин.

**Статус (2026-04-11):** таблица `public_profile_reports`, POST `music/report-profile` (auth, throttle) + форма на публичных профилях (`simple-entity`, `musician`); поле `moderation_review_requested_at` на публичных сущностях и в MoonShine (`MusicModerationForm`). **2026-04-11 (доп.):** таблица `moderation_audits`, observer’ы для профилей и жалоб, просмотр в MoonShine (`ModerationAuditResource`).

### Фаза 4 — Магазины и товары (универсально, new/used) (**ядро в коде; витрина и каталог — 2026-04-10**)

- Магазин: отдельная сущность (как студия/репточка/школа), 1 пользователь может иметь несколько магазинов (и других сущностей).
- Товары: **`good_id` обязателен**; ключ строки в магазине — `code` (артикул); импорт XLS/XLSX с привязкой к каталогу (`good_code` или совпадение с `goods.code`).
- New vs used:
  - new: фото по умолчанию из базы товара;
  - used: свои загруженные фото (приоритет), затем (опционально) фото из базы.
- **Категории на витрине:** из **`Good` → `categories`**; фильтр на публичной странице магазина `?category={id}` (`PublicMusicProfileController::shop`). Отдельная taxonomy только для позиций без каталога не делалась (все позиции с каталогом).
- Остатки: импорт/экспорт через XLS/XLSX; загрузка по `code`.

### Фаза 5 — Корзина/заказы/самовывоз (**частично в коде**)

- Корзина: только для авторизованных.
- В корзине могут быть позиции разных магазинов, но правило оформления: **1 заказ = 1 магазин** (`ShopCheckoutService`).
- Статус **«подтверждён магазином»** — `ShopOrderStatus::StoreConfirmed`.
- Доставка v1: самовывоз без обязательного адреса на заказе (адрес на заказе не введён в модель).

### Фаза 6 — Оплата (максимально общий слой) (**слой данных и ручной учёт — 2026-04-10; PSP нет**)

- В БД: **`ShopPaymentStatus`**, **`ShopPaymentMethod`**, суммы заказа, комиссия (`config/shop.php` + optional `shops.platform_fee_rate`), `paid_at`, `payment_external_reference`.
- **Интеграции с платёжками пока нет** — в MoonShine вручную перевод в «оплачен»; уведомление в ленту `notice` мессенджера (`ShopOrderMessengerNotifier`, `ShopOrderObserver`).
- Возвраты: не требуются в v1.

### Фаза 7 — Доставка (на вырост)

- Предусмотреть, что магазин может добавлять способы доставки; интеграция расчёта — позже.

**Статус (2026-04-11):** в `shop_orders` — `delivery_mode` (`pickup` | `shipping`) и `shipping_address_id` (FK на `addresses`, владелец — пользователь-покупатель); оформление в `ShopCheckoutService` и UI корзины; геокодинг и тарифы — позже.
