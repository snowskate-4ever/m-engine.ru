# Дальнейшее развитие музыкального сервиса (дорожная карта)

Обновлено: 2026-04-08.  
Документ дополняет закрытый чеклист в `music-ecosystem-plan.md` (п.1–11) и фиксирует **следующие эпики** без обязательства к срокам.

---

## Контекст

Базовый контур экосистемы реализован: публичные профили, кабинет, состав исполнителей с приглашениями, каталог для авторизованных, адреса, дублирование приглашений на e-mail. Ниже — направления, которые усиливают продукт для пользователей и операций.

---

## 1. Центр in-app уведомлений

**Зачем:** записи в `notifications` уже создаются (например, приглашение в состав), но без единой точки входа пользователь может их не замечать.

**Состав работ:**

- UI в шапке / боковой зоне: счётчик непрочитанных, выпадающий список последних событий.
- Страница «Все уведомления» с пагинацией, фильтром по типу (по `type` / данным payload).
- Пометка прочитано / «прочитать всё»; при ответе на приглашение — уже есть сброс в сервисе, проверить согласованность.
- Локализация заголовков и текста по типам уведомлений (в т.ч. `PerformerLineupInvitationNotification`).

**Зависимости:** нет жёстких; можно делать параллельно с другими пунктами.

---

## 2. Публичный каталог (без обязательной авторизации)

**Зачем:** сейчас `music/discover` доступен только авторизованным; гости и поисковики не видят «витрину» сообщества.

**Состав работ:**

- Публичный маршрут (например `/discover` или `/music`) с тем же или упрощённым поиском, что `MusicPublicSearchService`.
- Ограничения: только сущности с `public_page_enabled` и валидным `slug` (как сейчас в сервисе).
- Layout: `public-minimal` или лёгкая оболочка с входом в кабинет; без дублирования всего app-sidebar.
- Опционально: мета-теги / базовый SEO для страницы каталога.

**Зависимости:** переиспользование `MusicPublicSearchService`; политики авторизации не требуются для чтения публичных данных.

---

## 3. Администрирование в MoonShine (или отдельный admin)

**Зачем:** операционная поддержка, модерация «сухим» способом, обход проблемных записей без SQL.

**Состав работ:**

- Ресурсы или расширение существующих: `Musician`, `Teacher`, `Peformer`, `Studio`, `Rehersal`, `School` — просмотр, мягкое отключение публикации, правка slug при конфликтах.
- Связанные `Address` для музыких сущностей (полиморфный владелец уже есть в модели `Address`; в MoonShine MorphTo может потребовать расширения типов).
- Роли: доступ только админам MoonShine, без смешения с пользовательскими политиками `Gate`.

**Зависимости:** решение, что правится только из админки, а не дублирует весь кабинет.

---

## 4. Интеграции и дополнительные каналы уведомлений

**Зачем:** в спецификации п.8 заложены «чат, уведомления» и расширение на контакт; сейчас есть сайт + БД + почта.

**Возможные направления:**

- **Мессенджер:** при приглашении создавать системное сообщение / карточку в существующем чате (если есть общий канал с пользователем) — продуктово сложнее, нужны правила (только при существующем диалоге или всегда «бот»).
- **Telegram / VK:** опциональная привязка аккаунта и отправка короткого уведомления (в проекте уже есть зачатки VK у пользователя — оценить переиспользование).
- Единый **шлюз уведомлений** (какие каналы включены в `notification_preferences` наряду с `music_lineup_email`).

**Зависимости:** политика приватности, лимиты внешних API, очереди.

---

## 5. Модерация публичного контента

**Зачем:** в плане зафиксировано «модерации пока нет»; при росте сообщества понадобится контроль.

**Состав работ:**

- Флаги / жалобы от пользователей или скрытие по решению модератора.
- Состояния профиля: например «на проверке», «скрыт администратором» (отлично от «владелец выключил публикацию»).
- Админ-очередь в MoonShine или отдельном разделе.

**Зависимости:** юридические формулировки, хранение причин блокировки, аудит действий.

## 6. Качество и сопровождение

- **Тесты:** feature-тесты для `PerformerMembershipService`, публичного поиска, создания адреса (критичные сценарии).
- **Наблюдаемость:** логирование ошибок отправки почты, метрики по приглашениям.
- **Документация:** обновить блок «Порядок работ» в `music-ecosystem-plan.md` (чеклист закрыт; сослаться на этот файл как на roadmap).
